# Resume OS — Operating Model

This file explains how the resume project works. It is stable operating context, not the latest project state.

For current status, locked decisions, open questions, and next steps, read `resume-project-tracker.md`.

## File Roles

> **Layout note (multi-tenant):** engine files live at the repo root / `engine/`; per-tenant files live
> under `profiles/<activeProfile>/`. Specifically: base resumes are in `profiles/<activeProfile>/base-resumes/`,
> private source material in `.../sources/`, generated/pipeline state (`inbox/`, `applications/`, `events/`,
> `resume-formats/`, `jobs-tracker.md`, `package-queue.md`) in `.../work/`, and profile memory
> (`resume-project-tracker.md`, `LEARNINGS.md`, `review-schedule.md`, `positioning.md`) in the profile root.
> Tooling resolves these through `engine/config.mjs`; bare filenames below resolve within the active profile.
> See `README.md` for the full layout.

- `resume-project-tracker.md` — authoritative project memory: current state, locked decisions, open items, file relationships, and next steps.
- `resume.md` — canonical senior/general base resume for senior mass-apply and general sharing.
- `resume-pm.md` — reusable non-senior AI-enabled workflow PM base resume; removes pre-master's experience/education and avoids Senior PM headline positioning.
- `resume-po.md` — tactical Product Owner base resume for PO-labeled opportunities; keeps Senior PM as the primary identity while showing requirements clarity, backlog prioritization, cross-functional influence, delivery discipline, and measurable outcomes.
- `resume-ai-workflow.md` — reusable senior AI workflow / agentic systems base for AI agents, AI productivity, AI tooling, evals, and human-in-the-loop automation opportunities.
- `resume-commerce.md` — reusable senior commerce/ecommerce/retail 1SZFA variant for low-information commerce, retail-tech, marketplace, catalog, or merchandising opportunities.
- `resume-commerce-pm.md` — reusable non-senior commerce/ecommerce/retail PM 1SZFA variant; keeps the IC PM framing while foregrounding merchandising, catalog, and product-onboarding work.
- `exhaustive-experience.md` — private experience source of truth: every role, parked bullets, caveats, raw detail, interview prep, and tailoring stories. Never submitted.
- `skills-bank.md` — broad skills inventory and variant sets. It can contain candidate skills/tools that are not currently on the resume.
- `linkedin-profile-draft.md` — LinkedIn working copy; must remain a superset of resume content.
- `inbox/` — imported job source data. Each discovered job lives under `inbox/<job-id>/` with `job.md` for the readable JD and `metadata.json` for scraped facts plus lifecycle state.
- `jobs-tracker.md` — generated human-facing job board rendered from `inbox/*/metadata.json`. Do not hand-edit it.
- `applications/` — one-time application package snapshots. These are not base resumes and should not be regenerated by default base-export workflows.
- `scripts/job-board.mjs` — job lifecycle command. Use it to render the board and mark jobs `package-ready`, `applied`, `skipped`, or updated with outcomes.
- `prompts/claude-cowork-gmail-job-monitor.md` — daily Cowork/Gmail prompt for producing job email event files under `events/pending/`.
- `bullet-rubric.md` — bullet and whole-resume quality rules; apply before changing content.
- `tailoring-methodology.md` — the full tailoring engine: phased process, taste rules, feature-scorecard ship check, and export verification. Load this before any tailoring session.
- `eval-rubric.md` — the tailored-resume eval: keyword decision table, hard/soft/report ship gates, latent checklist. Deterministic checks run via `scripts/score-resume.mjs`.
- `model-change-evals-plan.md` and `evals/smoke/eval-instructions.md` — V0 smoke-test process for comparing LLM model changes using saved outputs and deterministic code checks.
- `adapters/claude-code-bootstrap.md` — Claude Code session bootstrap (runtime adapter); calls the resolver. (`cold-start-prompt.md` is a retired pointer to it.)
- `engine/resolver.md` + `engine/resolver.json` — routing table: task type → which docs to load (run `node scripts/test-resolver.mjs` to check). `engine/models.json` — per-step model config.

## Core Rules

- Resume content must stay honest and defensible: never inflate title, metric, scope, tool, or ownership.
- Rule: resume <= LinkedIn <= reality.
- Do not relitigate locked decisions unless the user explicitly asks.
- Do not change locked resume content without proposing the change and waiting for approval.
- Hard rule: do not undermine or retire the ready-to-send `resume-formats/` PDFs/HTMLs (the 1SZFA-style resumes). They are active general-purpose resumes for recruiter InMail, hiring events, email requests, and low-information or stretch applications where full tailoring is not warranted. Retired source downloads like `master-resume_1SZFA.md` are different from these active generated outputs.
- Keep the OS tracker updated at checkpoints; it is the memory across resume-system sessions. Job application lifecycle updates belong in `inbox/<job-id>/metadata.json` via `scripts/job-board.mjs`, with `jobs-tracker.md` regenerated as the view.
- **Working-data location (all agents):** every generated or pipeline artifact (job discovery in `inbox/`, Gmail/job tracking in `events/`, application packages in `applications/`, generated resumes in `resume-formats/`, plus `jobs-tracker.md` and `package-queue.md`) MUST be written under `profiles/<activeProfile>/work/`, never the repo root. This applies to every agent and external job (scrapers, the Gmail monitor, job discovery). Root copies are gitignored and invisible to the tooling, which resolves these paths via `engine/config.mjs`.
- **Root-memory boundary:** root `memory/` files record Resume OS engine architecture, durable decisions, and engine-only scratch state. They must never contain a candidate's job, application, outreach, interview, employer, resume-content, or pipeline details. Put all such state under `profiles/<activeProfile>/`—normally `work/` for pipeline artifacts, and the profile's tracker/source files for candidate-specific notes.
- Be concise, opinionated, and willing to push back when a claim or wording is weak.

## Collaboration Guardrails

- Use the user's exact wording when they specify a word or phrase. If a different word seems stronger, propose it instead of silently substituting it.
- Locked content and locked decisions are frozen unless the user explicitly reopens them.
- Re-read the relevant source files before asking the user for information. Do not ask for details that already exist on disk.
- Lead with the decision. For wording edits, show the exact change as `old wording -> new wording` and identify the affected bullet, section, or file.
- Do not silently reverse earlier decisions. If new information makes a locked choice questionable, flag the conflict and let the user decide.
- Park cut material in `exhaustive-experience.md` with pull-in guidance and caveats. Do not discard useful source material.
- Apply the same defensibility standard to links, projects, named tools, and side projects that you apply to employment bullets.

## Content Model

- Living base resumes are carved from `exhaustive-experience.md`.
- Reusable base resumes are allowed when the audience, seniority, or story changes structurally. Current bases: `resume.md` for senior/general roles, `resume-pm.md` for non-senior AI-enabled workflow PM roles, `resume-po.md` for PO-labeled opportunities, `resume-ai-workflow.md` for senior AI workflow / agentic systems opportunities, `resume-commerce.md` for senior commerce/ecommerce/retail opportunities, and `resume-commerce-pm.md` for non-senior commerce/ecommerce/retail PM opportunities.
- Application packages are disposable snapshots. Create new ones under `applications/<Company - Role>/`. The detailed process lives in `tailoring-methodology.md`; this file only records the convention.
- Current packages usually keep `job.md`, `strategy.md`, `keywords.md`, `resume.md`, `resume.html`, a single human-facing upload PDF, and optional `tailoring-log.md` / `cover_letter.md` / `answers.md` at the package root. The upload PDF filename must start with `<Full Name>`, e.g. `<Full Name> - <Company Role> - Resume.pdf`. There is no separate generic `resume.pdf` (dropped 2026-06-15; tooling scores the temp export dir, not the package).
- Job lifecycle state lives with the imported job, not the application package: update `inbox/<job-id>/metadata.json` through `node scripts/job-board.mjs`. `jobs-tracker.md` is a generated board, so manual edits there will be overwritten on render.
- The generated board separates active streams: `Applied` is submitted/waiting, `Needs Action` is follow-up required, `Interviewing` is screen/interview activity, and `Closed` is final outcomes such as rejection or withdrawal.
- Email tracking should use event handoff files, not direct tracker edits. Cowork/Claude should use `prompts/claude-cowork-gmail-job-monitor.md` to write read-only Gmail findings into `events/pending/`; this repo imports those events into lifecycle metadata and archives the event files to `events/archive/`. **TODO:** `scripts/import-events.mjs` (deterministic import) is not yet built — event import is currently manual (see `resume-project-tracker.md` open questions).
- Event files are immutable. Review/import status belongs in `events/reviews.md` with reviewer, model, timestamp, decision, and notes; do not edit old event blocks to mark them reviewed.
- Older packages may only contain generated HTML/PDF or replies. Leave them with their company folders as historical records; package conventions guide new work but are not a cleanup mandate.
- Submitted application packages are frozen. Do not edit their resume, PDF, HTML, cover letter, answers, or notes unless the user explicitly says the submitted package is reopened or asks to amend that exact package. Improvements after submission go into base resumes, reusable variants, or future packages only.
- Artifact-boundary rule: base resumes and application packages do not auto-propagate into each other. Updating `resume.md`, `resume-pm.md`, `resume-po.md`, `resume-ai-workflow.md`, `resume-commerce.md`, `resume-commerce-pm.md`, or `resume-formats/` never implies regenerating existing packages. Updating a package never implies changing base resumes. Cross the boundary only when the user explicitly asks for that direction.
- Do not create a new reusable base for every job posting; job-specific variants belong in tailoring.
- Base routing precedence: commerce/retail/ecommerce/marketplace domain wins base selection when clearly dominant; PO title or PO-framed delivery roles default to `resume-po.md` unless a domain-specific base is clearly stronger; AI wins emphasis. AI commerce roles start from the commerce base, then promote AI workflow proof during tailoring. Use `resume-ai-workflow.md` when the core product surface is AI agents, agentic workflows, AI tooling/productivity, evals, or human-in-the-loop automation.
- Tailoring is separate and on-demand: pull from parked bullets and backup stories only when a job posting makes them more relevant.
- Resume bullets should be simple, scannable, defensible, and conversation-inviting.
- Do not justify bullet text. Keep bullets left-aligned; improve readability through margins, line height, font size, bullet spacing, and line measure instead.
- Depth belongs in interview prep, backup stories, and tailoring notes, not in overloaded resume bullets.
- LinkedIn should be broader than the resume so tailored variants remain corroborated publicly.

## Cover Letter Skill

Use this when an application asks for a cover letter or the user explicitly wants one. The cover letter is a small application artifact, not a second resume.

- Read the package `job.md`, `strategy.md`, `keywords.md`, and tailored `resume.md` before drafting.
- Default shape: one short opener, 3-4 bullets that map real experience to the JD, one short close, then contact info. Use paragraphs only when the user asks for a more traditional letter.
- Keep it short unless the application asks for more. The letter should usually fit comfortably on one page and be easy to skim in under 30 seconds.
- Each bullet should connect one JD requirement to one defensible proof point: AI/ML delivery, analytics/experimentation, workflow/platform ownership, customer feedback, build-vs-buy judgment, domain motivation, or location/company fit.
- Use passion sparingly and specifically: connect it to the company's product problem or user world, not generic enthusiasm.
- Claims must come from the tailored resume, `exhaustive-experience.md`, `skills-bank.md`, or explicit user input. Reframe and emphasize; do not introduce new scope, tools, metrics, or domain experience.
- Preserve honest gaps. If the JD asks for gaming, DBT, Alation, healthcare, fintech, or another domain/tool the user has not done, bridge through adjacent evidence instead of claiming direct experience.
- Package artifacts: keep editable `cover_letter.md`, generated `cover_letter.html`, and a name-leading upload PDF such as `<Full Name> - <Company Role> - Cover Letter.pdf` beside the resume package.
- Render cover-letter PDFs with Chrome header/footer disabled (`--no-pdf-header-footer --print-to-pdf-no-header`); this is part of the cover-letter skill, not an optional cleanup step. After generation, verify the PDF is 1 page and does not show browser headers/footers such as file path, date, page number, or document title. Keep the style simple: candidate header, subtle rule, readable body, no decorative layout.

## Tailoring Discipline

- Tailoring is job-specific and separate from the canonical resume.
- Tailored resumes should actively use the variable surfaces: profile/headline wording, selective bolding/emphasis, and the skills section. These are tailoring levers, not fixed mass-apply defaults.
- For every tailored package, review and rewrite the fit summary/headline against the JD. It is a role-specific positioning surface, not boilerplate to inherit from the base or the last application.
- For AI/tooling roles, do not position the user as merely "a PM who uses tools and builds things." The stronger PM framing is: AI and technology shorten the path from high-friction workflow discovery to prototype, user feedback, validation, clearer specs, adoption, and change management.
- Use selective bolding sparingly: emphasize the main scan anchor in a bullet, usually the primary hard metric or the shipped product/system name. If everything is bold, nothing is.
- Prefer skills swaps, profile/headline emphasis, selective bolding, and bullet re-ranking before rewriting bullets.
- Pull extra bullets or proof points only from `exhaustive-experience.md`, `skills-bank.md`, or already-approved project material.
- Always scan `exhaustive-experience.md` before tailoring to a role; there may be stronger or more relevant parked bullets/stories than the default resume currently shows.
- Keep tailored variants within the 2-page bound. If adding skills, stronger proof points, or role-specific bullets, do a relevance pass and drop or compress the least relevant existing bullets for that posting.
- Rewrites must stay grounded in existing source material and remain interview-defensible.
- Confirm tailored claims are backed by LinkedIn, or update LinkedIn before relying on them in a submitted resume.

### Role Archetype Tailoring Pass (run this first)

For the full tailoring process with evaluation loop, scoring framework, and taste rules, load `tailoring-methodology.md`. The steps below are the core archetype translation — they're a subset of the methodology's Phase 1.

Before editing any bullet, translate the work into the company's language. Tailoring quality is mostly a function of how well you read the JD, not how many keywords you mirror. jobforge is being updated to automate this pass (PR #68, `role_intel` + `resume_tailor`); until that merges and deploys, do it by hand the same way:

1. **Extract the user archetype.** Who does this team build for: merchant, operator, seller, buyer, student, admin, developer? One line. Reframe only when the work genuinely maps to that audience; do not carry one company's vocabulary into unrelated applications.
2. **Build a vocabulary map.** For each problem the JD cares about, map their term to the candidate's real, evidenced experience. Only map a term the resume/`exhaustive-experience.md` genuinely supports — honest translation, never keyword insertion. The evidence anchor is the guardrail.
3. **Add one explicit discovery signal** when the JD values customer/user empathy — name who you talked to (retail staff, enterprise customers, students), woven into a role, not a generic standalone bullet.
4. **Promote builder proof into bullets** when the JD asks for coding/prototyping/AI tooling. Move it out of the skills list into experience or project bullets so it's visible, but frame the value as faster discovery, validation, sharper specs, adoption, and reduced waste before engineering commitment.
5. **High-fit / referral exception to the dropped profile section:** for a strong-fit or referred application, a 2-line tailored summary at the top is allowed even though base resumes carry no profile section. It frames the archetype before the reader parses bullets.
6. **Expand a current project** when it proves a core requirement (e.g. JobForge for agentic/AI-builder roles); **compress older, low-relevance roles** to fund the relevant proof while holding the 2-page bound.
7. **Named tools stay defensible** — add a tool only if confirmed in `skills-bank.md` and personally backed (e.g. Cursor is confirmed; add it for AI-tooling roles that name it).

## Living Base Resumes

Every profile has a small set of untailored base resumes — currently the senior/general `resume.md`, the non-senior AI-enabled workflow PM `resume-pm.md`, the tactical Product Owner `resume-po.md`, the senior AI workflow / agentic systems `resume-ai-workflow.md`, the senior commerce/ecommerce/retail `resume-commerce.md`, and the non-senior commerce/ecommerce/retail PM `resume-commerce-pm.md`. These are first-class, required artifacts, not disposable ones, and the OS is designed to generate and maintain them:

- It is carved from `exhaustive-experience.md` and kept current as the canonical, high-quality general resume for that profile.
- It is what you send when tailoring is not possible or not warranted: a recruiter's LinkedIn InMail, an email request, or someone you meet at an event. It must stand on its own and capture the most important things without a specific JD.
- The generated `resume-formats/` outputs are the ready-to-send versions of these bases, not stale clutter. Keep them available and refreshed after base edits.
- It is the base that feeds jobforge (`data/resume.md` / `users.resume`); jobforge tailors *from* it per job into `applications/` snapshots. Tailoring is jobforge's job; producing and maintaining the general base resume is the OS's.

Keep sources intentional and few. Stale or random-named source downloads of a base (old GPT downloads, etc.) are not additional resumes — reconcile any unique content back into `exhaustive-experience.md` or the relevant base, then discard the duplicate source. Do not apply that cleanup rule to active generated outputs under `resume-formats/`.

## ATS / Submission Checks

- Content checks happen on the source resume Markdown (`resume.md`, `resume-pm.md`, `resume-po.md`, `resume-ai-workflow.md`, `resume-commerce.md`, `resume-commerce-pm.md`, or a tailored variant) or the generated HTML; rendered-file checks happen on the exported PDF.
- Before broad submission, run a PDF parse check: export the PDF, select all, copy, paste into plain text, and confirm the content is complete and in reading order.
- Keep resume formats single-column and text-based. Avoid tables, text boxes, columns, and image-rendered text.
- Generated senior/general formats are built from `resume.md`; after content edits, run `node scripts/build-resume-formats.mjs --export` to refresh active HTML/PDF/PNG variants as `<Full Name> - Senior Product Manager`.
- Generated PM formats are built from `resume-pm.md`; run `node scripts/build-resume-formats.mjs --source resume-pm.md --variant pm --export` to refresh `resume-formats/pm/` as `<Full Name> - Product Manager`.
- Generated Product Owner formats are built from `resume-po.md`; run `node scripts/build-resume-formats.mjs --source resume-po.md --variant po --export` to refresh `resume-formats/po/` as `<Full Name> - Product Owner`.
- Generated senior AI workflow formats are built from `resume-ai-workflow.md`; run `node scripts/build-resume-formats.mjs --source resume-ai-workflow.md --variant ai-workflow --resume-title "Senior Product Manager - AI Workflow" --export` to refresh `resume-formats/ai-workflow/`.
- Generated senior commerce formats are built from `resume-commerce.md`; run `node scripts/build-resume-formats.mjs --source resume-commerce.md --variant commerce --resume-title "Senior Product Manager - Commerce" --export` to refresh `resume-formats/commerce/`.
- Generated non-senior commerce PM formats are built from `resume-commerce-pm.md`; run `node scripts/build-resume-formats.mjs --source resume-commerce-pm.md --variant commerce-pm --resume-title "Product Manager - Commerce" --export` to refresh `resume-formats/commerce-pm/`.
- Format configuration lives in `resume-format.config.json`. `emitNarrow` is currently `false`, so normal builds produce only the default ruled resume. Use `--with-narrow` for a one-off narrow export if needed.

## Positioning Frame

Positioning is candidate-specific, so it lives with the active profile, not in this reusable skill. Load `profiles/<activeProfile>/positioning.md` (and the `positioningFrame` block in `profiles/<activeProfile>/profile.json`) before writing headlines, summaries, project bullets, LinkedIn, cover letters, or interview prep.

A profile's positioning frame should define, per candidate:

- **Narrative spine:** the cross-role thread that ties the experience together — the kind of problems this person turns into products and the judgment behind it. Lead with whatever the role rewards, but tie it back to product judgment rather than a tool list. Avoid framing the candidate solely as a "growth PM" when the real story is better workflows, lower friction, and improved unit economics.
- **AI / domain usage frame (when relevant):** how the candidate uses AI/technology as product leverage — framed as faster learning, sharper specs, and validated direction, not "uses AI tools to build things." Cover discovery, problem framing, prototyping, validation, specs/delivery, prioritization, GTM/change management, and quality/governance.

Use the profile's frame to keep the resume from collapsing into a generic noun/tool dump.

## Skills Philosophy

The skills section is not the main proof of seniority. The bullets carry the proof.

Skills on the resume should do three jobs:

1. ATS coverage: include terms the bullets already prove.
2. Narrative reinforcement: make the cross-role pattern obvious.
3. Tailoring surface: provide a clean place to swap keywords per job without rewriting the whole resume.

Avoid treating the resume skills section as a generic noun dump. A skills section full of generic claims can read like filler because anyone can list them. Do not add skills just because they might pass ATS; if the term has no clear story or credibility, keep it off the resume.

`skills-bank.md` should preserve the natural inventory of skills and tools. The resume should pull a targeted subset from that bank based on the narrative, role target, and interview defensibility.

Named tools need stricter scrutiny than broad capabilities. A named tool belongs on a resume only if there is a concrete story for what the user personally used, owned, shipped, configured, analyzed, or directed with it.

## Skills Bank vs Resume Skills

- `skills-bank.md` can be broad and include candidate skills/tools for future tailoring.
- `resume.md` should be narrower and evidence-backed.
- Variant skills sets are useful for high-match roles, but the default resume should not try to cover every possible direction.
- If a skill is only weakly supported, keep it in the bank until a concrete story validates it.
- The default resume skills block should stay compact and credible: Product, AI & Automation, Agentic Development Stack, and Technical & Analytics. `AI & Automation` describes product areas and systems; `Agentic Development Stack` names hands-on AI-builder tools used for prototyping and agentic product work.
